- FIX REST `/set` endpoint not running `changeValidator` for new documents. When a client sent a document whose primary key did not yet exist on the server, the insert path skipped the `changeValidator` entirely while the update path still validated the change. A `changeValidator` configured to authorize writes was therefore bypassed for new inserts, allowing clients to create documents that an update of the same document would not be allowed to produce. The handler now invokes `changeValidator` for new inserts with `assumedMasterState: undefined`, aligning the behavior with the replication `/push` endpoint and rejecting disallowed inserts with 403 Forbidden.
